# Subme 前端重塑方案 —— 「淬钢 QUENCH」

> 一句话定位：把「API 转售站的管理页面」升级为「**一家公司的经营操作台**」。
> 设计基调：**淬火中的冷钢** —— 冷石墨基底、金属渐变面、淬火蓝光晕只给焦点与活性、等宽数字、专业级信息密度。
> 交互式视觉稿：`docs/design/mockup-quench.html`（浏览器直接打开，含驾驶舱 / 用户管理 / 账号池 / 登录页四屏）。
> 历史方向存档：方向 A「鎏金 AURIC」（暖金）见 `mockup-auric.html`，已被本方向取代，仅留作对比。

## 一、诊断：为什么现在的前端「不适合运营」

不是审美问题，是**视角错位**——现在的前端是「数据库表的镜像」，而运营需要的是「业务实体的操作台」。

| 病灶 | 证据 | 运营痛点 |
|------|------|----------|
| 巨型单文件 | `SettingsView.vue` 9784 行、`GroupsView.vue` 4343 行 | 任何改动都是高风险手术，无法拓展 |
| 记录视角，无实体视角 | `UsersView.vue` 只是一张表，无用户详情页 | 看一个用户的订阅+余额+Keys+订单要跳 5 个页面 |
| 商业化碎片化 | 套餐 / 订单 / 卡密 / 优惠码 / PayGo 散在 5 个一级菜单 | 没有统一的「收入账本」，定价靠记忆 |
| 账号=普通 CRUD | `AccountsView.vue` 1729 行的增删改查表（deprecated, 已由 `AccountsPoolView.vue` 取代，见 Phase 4 迁移） | 看不到供给容量、健康度、成本，无法做容量规划 |
| Modal 地狱 | 所有编辑都是全屏弹窗，无上下文 | 改一个字段丢失列表位置，批量操作低效 |
| 无视图记忆 | 筛选条件不可保存、不可分享 | 「余额告急的付费用户」这种日常视图每次重新拼 |
| 信息架构按表组织 | 导航 = 数据库表名清单 | 新人无法从导航理解业务 |

**底子的优点（保留）**：Vue 3 + Vite + Tailwind + reka-ui 无头组件、`DataTable.vue` 已有 15+ 页复用、API 层有 axios 拦截器与 token refresh、i18n 与暗色模式已就位。原主题「Cold Steel」的冷钢初心与本方案同脉，但执行粗糙——**方案是渐进重塑，不是推倒重写。**

## 二、设计语言：「淬钢 QUENCH」

### 2.1 概念

这是一个**管钱、管供给、管客户**的系统——一座精密的冷锻车间。设计隐喻取「淬火钢材」：

- **冷钢**：深冷石墨基底（去暖去棕，纯冷调），金属渐变面 + 顶部 1px 内高光，沉稳、专业、低疲劳；
- **淬火蓝光**：唯一的高能色是淬火蓝 `#5CA8FF`，**只出现在三处：焦点、活性、实时**——输入框聚焦的光晕环、当前导航、LIVE 数据。蓝光亮起的地方就是系统正在响应你的地方；
- **账本**：所有数字一律等宽字体 + tabular-nums，右对齐；金额用最亮的钢白并带极淡冷光晕——在一片冷灰中，钱是最亮的东西。

层级哲学：**底色退后、金属面浮起、蓝光聚焦**。三层光线关系替代传统的色块堆砌。

### 2.2 设计令牌（Design Tokens）

```css
:root[data-theme="dark"] {
  /* 基底：冷石墨（去蓝灰偏移，纯冷黑） */
  --bg-0: #08090B;        /* 页面底 */
  --bg-1: #101216;        /* 卡片 */
  --bg-2: #171A20;        /* 浮层 / hover */
  --line-0: #20242C;      /* 分隔线 */
  --line-1: #2F3540;      /* 强边框 */

  /* 墨色：冷白 */
  --ink-0: #E8EBF0;       /* 主文字 */
  --ink-1: #97A0AF;       /* 次级 */
  --ink-2: #5C6470;       /* 弱化 */

  /* 淬火蓝：只用于 焦点 / 活性 / 实时 */
  --azure: #5CA8FF;
  --azure-dim: rgba(92,168,255,.12);
  --glow-focus: 0 0 0 1.5px rgba(92,168,255,.65), 0 0 20px rgba(92,168,255,.28);

  /* 金属面：卡片与按钮的渐变与边缘高光 */
  --metal:        linear-gradient(180deg, #15181E, #0E1014);
  --metal-raised: linear-gradient(180deg, #272D37, #14171D);
  --edge-hi: inset 0 1px 0 rgba(255,255,255,.06);

  /* 金额：钢白 + 冷光晕（全站最亮元素） */
  --money: #F2F5FA;
  --money-glow: 0 0 18px rgba(214,232,255,.22);

  /* 语义色：状态专用，绝不用于装饰 */
  --ok:   #46C98C;  --warn: #E0B34E;
  --bad:  #F25C69;  --info: #5CA8FF;

  --radius: 12px;          /* 卡片 12 / 输入与按钮 10 / 登录卡 20 */
  --shadow-card: var(--edge-hi), 0 12px 32px rgba(0,0,0,.5);
}
```

亮色主题同构（冷白纸面 `#F4F6F9` + 深钢字 + 同一支淬火蓝），通过 `data-theme` 切换，**令牌名不变**。深色为第一公民，亮色是降级适配。

### 2.3 字体

| 用途 | 字体 | 理由 |
|------|------|------|
| Latin 标题/导航 | **Archivo**（SemiExpanded 用于大标题） | 工业感、宽肩膀，与"操作台"气质一致，避开 Inter 俗套 |
| 数字/代码/金额 | **IBM Plex Mono** | tabular 数字、仪表气质；所有金额、token 数、ID 一律用它 |
| 中文 | 思源黑体 / PingFang SC 栈 | 与 Archivo 字重曲线匹配 |

规则：**金额永远是 `mono + 钢白光晕 + 右对齐`**——用户在任何页面扫一眼就知道钱在哪。

### 2.4 质感与氛围

- 背景：2% 噪点 + 极淡坐标纸网格 + 顶部一抹冷蓝环境光（radial），制造「车间顶灯」的纵深；
- 卡片：金属渐变面（上亮下暗）+ 1px 冷边框 + 顶缘 1px 内高光——是「锻面」不是「色块」；
- 按钮：主按钮用 `--metal-raised` 金属凸面（参考稿中的深色金属 CTA），hover 时边缘泛起淬火蓝微光；**不用大色块按钮**；
- 焦点态：`--glow-focus` 蓝光环（1.5px 实环 + 20px 光晕），输入框、行选中、键盘焦点统一此语言，全键盘可达；
- 状态点带呼吸动画（仅 `warn/live` 有动效，避免满屏闪烁）。

### 2.5 动效准则

- **一次编排好的入场** > 满屏微动效：页面加载时卡片以 40ms 阶梯延迟上浮淡入（`cubic-bezier(.2,.8,.2,1)`，280ms）；
- 关键数字入场跳动归位（counter tick），让「今日营收」有心跳感；
- 抽屉滑入 240ms，遮罩淡入 160ms；表格行 hover 出现淬火蓝左缘线（2px）；
- 光晕的呼吸（focus ring、LIVE 点）周期 ≥1.6s，缓慢而克制；
- 所有动效尊重 `prefers-reduced-motion`。

### 2.6 密度模式

运营后台必须支持 **comfortable / compact** 两档密度（行高 44px / 32px），全局切换，存 localStorage。表格默认 compact。

## 三、信息架构：按「业务域」重组，不按数据库表

```
┌─ 🏛 驾驶舱        营收·流量·异常 三总览（运营每天的第一屏）
├─ 👥 客户          用户（列表+用户360） · API Keys · 邀请返利
├─ 💰 商业化        套餐目录 · PayGo 计价 · 订单 · 卡密 · 优惠码
├─ ⚙ 供给          账号池 · 分组与路由 · 渠道定价 · 代理
├─ 🛡 可靠性        实时监控 · 告警 · 错误日志 · 风控
└─ 🔧 平台          设置 · 公告 · 自定义页 · 备份 · 审计日志
```

对照现状的关键变化：

| 现状 | 重组后 | 理由 |
|------|--------|------|
| 用户管理 = 一张表 | 客户域：列表 + **用户 360** | 运营操作以「人」为中心 |
| 套餐/订单/卡密/优惠码 各自为政 | **商业化** 一个域 | 钱的事情在一个屋檐下 |
| 账号/分组/渠道/代理 平铺 | **供给** 域，账号池为库存视角 | 这四者本来就是一条供给链 |
| Ops/监控/风控 分散 | **可靠性** 域 | SRE 心智模型 |
| 设置 = 9784 行单文件 | 平台域 + 设置注册表 | 见 §6.4 |

用户端（购买侧）信息架构基本健康，本期只做**视觉令牌统一**，不动结构。

## 四、核心交互范式（六式）

1. **主从抽屉（Master-Detail Drawer）**：列表页点击行 → 右侧 560px 抽屉展开详情/编辑，列表不丢位置。抽屉状态写入 URL（`?panel=user:123`），可深链、可刷新、可分享。复杂实体（用户 360）抽屉内可「展开为全页」。
2. **命令面板 ⌘K**：全局搜索用户/账号/订单/设置项 + 快捷动作（"给 user@x 充值"、"禁用账号 #42"）。运营的肌肉记忆入口。
3. **保存视图（Saved Views）**：筛选条件 + 列配置 + 排序 = 一个可命名视图，置顶为列表页签（如「余额告急」「7 日流失风险」「已封禁」）。视图可团队共享。
4. **批量操作条**：勾选行后底部浮出操作条（调组/调额/封禁/导出），带进度与逐项失败回执（现有 batch delete 的进度条模式推广到所有批量动作）。
5. **就地编辑（Inline Edit）**：余额、RPM、分组等单字段双击即改，optimistic update + 失败回滚 toast。小改动不开弹窗。
6. **实时脉搏**：驾驶舱与账号池接入现有 WS QPS 通道，关键指标 5s 心跳刷新；异常时侧边导航对应域出现红点脉冲。

## 五、关键页面方案

### 5.1 客户域 · 用户管理 + 用户 360 ⭐

**列表页**：
- 顶部保存视图页签：`全部 / 付费用户 / 余额告急 / 流失风险 / 已封禁`；
- 列：用户（头像+邮箱）、套餐徽章、**余额（mono 钢白光晕 + 余量条）**、30 日消耗 sparkline、Keys 数、状态、最后活跃；
- 行级快捷动作：充值、调组、封禁（hover 浮现）。

**用户 360 抽屉**（本方案的灵魂）：
```
┌──────────────────────────────────────┐
│ 头像  user@email   [付费][高消耗][↗展开全页] │
│ ──────────────────────────────────── │
│ 余额 $12.40 │ 本月消耗 $86.2 │ Pro 月付·23天 │  ← KPI 条
│ ──────────────────────────────────── │
│ 概览 | 订阅 | Keys | 订单 | 用量 | 风控      │  ← 页签
│                                      │
│ · 30日消耗曲线（按模型分层）              │
│ · 活动时间线（充值/订阅/告警/封禁 混排）     │
│                                      │
│ [充值] [赠送时长] [调整限速] [封禁]        │  ← 底部操作栏
└──────────────────────────────────────┘
```
后端已有全部 API（users / subscriptions / orders / usage / risk-control），这是**纯前端聚合**，零后端改动。

### 5.2 商业化域 · 套餐与 PayGo ⭐

一个「商业化中心」三屏：

1. **套餐目录**：套餐卡片墙（直接复用用户端购买页的卡片视觉，做到「所见即所卖」），每卡片叠加运营数据：订阅数、月收入、转化率；拖拽排序 = 售卖页顺序。
2. **PayGo 计价台**：
   - 价格矩阵：`模型 × 分组` 表格，单元格 = 实际单价（渠道底价 × 分组系数），就地编辑系数；
   - **毛利透视**：每行同时显示「上游成本 / 售价 / 毛利率」，毛利率<0 的格子直接标红——现在定价全靠心算，这是运营最缺的一张表；
   - 计价模拟器：选模型+输入 token 量 → 实时算出各分组下用户付多少、你赚多少。
3. **收入流水**：订单 + 卡密核销 + 退款统一时间线，按 渠道/套餐/支付方式 分面过滤。

### 5.3 供给域 · 账号池 ⭐

把账号从「表格」升级为「**库存仪表**」：

- 顶部供给总览条：总账号数 / 健康 / 限流 / 耗尽 / 封禁，+ 总容量利用率；
- 主视图双模式切换：
  - **健康矩阵**（默认）：每账号一张紧凑卡片，按分组列阵——渠道图标、利用率环（淬火蓝）、RPM 实时值、状态呼吸点、成本/日。一屏看全供给侧水位，限流账号橙色脉冲，耗尽变灰；
  - **表格模式**：沿用 DataTable 做批量运维（换代理、刷新凭证、调组）；
- 容量预警：当「分组内健康容量 < 近 7 日峰值消耗 × 1.2」时分组卡片亮黄牌——把容量规划从事故驱动变成水位驱动。

### 5.4 驾驶舱

运营每天的第一屏，三段式：

1. **经营行**：今日营收（钢白大数字+counter tick+冷光晕）、MRR、PayGo 消耗、新增付费、ARPU——全部 mono 钢白；
2. **流量行**：实时 QPS（WS·淬火蓝 LIVE）、错误率、Top 模型消耗、渠道切换率；
3. **异常行**：余额告急用户 Top5、限流账号、未处理告警、可疑风控事件——每项可点击直达对应域并自动套用筛选。

### 5.5 认证页（登录/注册）

参考稿气质的完整还原（见视觉稿 `#login`）：

- 全屏冷黑 + 顶部环境光晕，居中 420px 金属锻面卡（radius 20px）；
- Logo 银白金属标 + 站名 + 一句「登录到控制台」；
- 输入框聚焦时亮起淬火蓝光晕环（`--glow-focus`）——整个页面唯一的高能时刻；
- CTA 是金属凸面按钮，不是色块；底部弱化的邀请码注册链路。

## 六、工程架构：为「易于拓展」而设计

### 6.1 目录：按 feature 切分

```
src/
├─ design/            # 设计系统包：tokens.css · 原子组件 · 密度系统
├─ features/
│   ├─ customers/     # api/ model.ts resource.ts views/ components/
│   ├─ monetization/
│   ├─ supply/
│   ├─ reliability/
│   └─ platform/
├─ shared/            # DataTable v2 · DetailDrawer · CommandPalette · SavedViews
└─ app/               # router · shell(侧边栏/顶栏) · providers
```

### 6.2 资源描述符：80% 的 CRUD 页不再手写

绝大多数管理页 = 表格+筛选+表单+批量操作。把它声明化：

```ts
// features/supply/proxies/resource.ts —— 整个代理管理页就这一个文件
export const proxyResource = defineResource({
  key: 'proxies',
  api: proxyApi,
  columns: [
    { key: 'name', title: '名称', sortable: true },
    { key: 'protocol', title: '协议', render: 'badge' },
    { key: 'latency', title: '延迟', render: 'mono', align: 'right' },
    { key: 'status', title: '状态', render: 'status' },
  ],
  filters: [ { key: 'protocol', type: 'select' }, { key: 'status', type: 'select' } ],
  form: proxyFormSchema,            // zod schema → 自动渲染抽屉表单
  bulkActions: ['delete', 'test'],
  rowActions: ['edit', 'test', 'delete'],
})
```

`<ResourcePage :resource="proxyResource" />` 自动获得：表格、筛选、分页、保存视图、抽屉表单、批量操作条、URL 状态同步。**新增一个管理页 ≈ 100 行描述符**；用户 360、账号池这类明星页面才写自定义视图。

### 6.3 状态分层

- **服务端状态**：引入 `@tanstack/vue-query`（缓存、重试、optimistic update、轮询统一收口），替代现在各页面手写的 loading/error/refresh 样板；
- **会话状态**：Pinia 只留 auth / 主题 / 密度 / 命令面板；
- **URL 状态**：筛选、分页、抽屉 panel 全部进 query string——可分享是运营协作的基本盘。

### 6.4 拆弹：SettingsView 9784 行 → 设置注册表

```ts
registerSettingsSection({
  id: 'payment.stripe', domain: '平台', title: 'Stripe',
  schema: stripeSettingsSchema,     // zod → 自动表单
  component: undefined,             // 特殊交互才提供自定义组件
})
```

设置页变成「注册表渲染器」：左侧锚点目录 + 全文搜索（接 ⌘K），每节一个 schema 文件（约 50-150 行）。新增配置项 = 加一个 section 文件，**不再碰万行巨石**。

### 6.5 防腐守则（写进 lint/CI）

1. 单个 `views/**/*.vue` 超过 **300 行** 报警，超过 500 行 CI 拒绝；
2. 颜色/间距/圆角禁止裸值，必须走 token（stylelint 规则）；
3. 金额/数字列必须 `mono + tabular-nums`；
4. 淬火蓝只允许出现在 焦点/活性/实时 语义，禁止装饰性使用（design review 检查项）；
5. 新增管理页默认走 `defineResource`，写自定义视图需在 PR 说明理由。
